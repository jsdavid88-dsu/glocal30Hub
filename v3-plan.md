# 글로컬30 R&D Hub v3 — 구현 기준 설계 문서

**문서 목적:** 이 문서는 글로컬30 R&D Hub v3를 구현하기 위한 기준 설계 문서다. 다른 개발자, 다른 AI 에이전트, 이후 투입되는 협업자가 이 문서만 읽어도 제품의 목적, 권한 구조, 핵심 데이터 모델, 화면 구성, 구현 우선순위를 이해하고 개발을 시작할 수 있어야 한다.

**작성일:** 2026-03-10  
**버전:** v3 revised  
**성격:** 내부 운영용 시스템 설계 문서  
**목표:** 연구실, 교수, 학생, 외부업체가 함께 쓰는 운영 허브 구축

## 1. 제품 목표

글로컬30 R&D Hub v3는 연구 조직 운영을 위한 내부용 플랫폼이다.
핵심 목표는 아래와 같다.

- 프로젝트 단위 협업을 한 곳에서 관리한다.
- 학생 개별 활동과 프로젝트 활동을 동시에 추적한다.
- 교수는 지도학생 기준, 프로젝트 기준으로 상황을 파악할 수 있어야 한다.
- 외부업체는 자신이 참여한 프로젝트 범위 안에서 필요한 정보를 충분히 볼 수 있어야 한다.
- 데일리 작성은 최대한 귀찮지 않아야 하며, 나중에 프로젝트별/학생별/이슈별로 다시 구조화되어 조회되어야 한다.
- 일정, 태스크, 출결, 이슈, 성과물을 한 플랫폼에서 연결해 본다.
- 향후 SOTA, 자동 리포트, Google Calendar, LLM 요약 기능까지 확장 가능해야 한다.

이 시스템은 외부 판매용 MVP가 아니라 내부 실사용을 위한 제품이다. 따라서 기능 수를 억지로 줄이는 것보다, 실제 운영 구조를 반영하고 쓰기 쉽게 만드는 것이 더 중요하다.

## 2. 주요 사용자

### 2.1 최고관리자
- 전체 시스템을 관리한다.
- 모든 사용자, 프로젝트, 데일리, 태스크, 출결, 일정, 리포트를 열람 및 수정할 수 있다.
- 역할 부여, 지도교수 지정, 프로젝트 권한 조정이 가능하다.

### 2.2 교수
- 자기 지도학생의 활동을 추적한다.
- 자신이 참여 중인 프로젝트를 관리하거나 열람할 수 있다.
- 지도학생 기준으로도 보고, 프로젝트 기준으로도 볼 수 있어야 한다.
- 프로젝트에 직접 참여하지 않은 경우, 그 프로젝트 전체 정보는 기본적으로 볼 수 없다.

### 2.3 학생
- 자신의 데일리, 태스크, 일정, 출결을 관리한다.
- 참여 중인 프로젝트의 관련 정보에 접근할 수 있다.
- 프로젝트 없는 개인 업무도 기록할 수 있다.

### 2.4 외부업체 사용자
- 참여 중인 프로젝트에 한해서 관련 데일리, 태스크, 이슈, 성과물을 열람할 수 있다.
- 프로젝트 외 학생 개인 정보, 행정성 정보, 장학금 관련 기록 등은 열람할 수 없다.

## 3. 권한 설계 원칙

이 시스템의 권한은 단순 role 기반이 아니라 role + relation 기반으로 설계한다.

즉, 아래 두 가지를 같이 본다.

- 사용자 역할(role): admin / professor / student / external
- 관계(relation): advisor_of / member_of_project / lead_of_project

### 3.1 중요한 원칙
- `교수`와 `프로젝트 리드`는 같은 개념이 아니다.
- 프로젝트 리드는 교수, 학생, 외부업체 사용자 누구든 될 수 있다.
- 교수는 지도학생을 기준으로 데이터를 볼 수 있다.
- 프로젝트 데이터는 프로젝트 참여 관계를 기준으로 본다.
- 최고관리자는 전체 권한을 가진다.

### 3.2 기본 권한 규칙

#### 최고관리자
- 모든 데이터 열람 가능
- 모든 데이터 수정 가능
- 사용자 역할 변경 가능
- 지도교수 관계 설정 가능
- 프로젝트 권한 설정 가능

#### 교수
- 자기 지도학생의 데일리, 일정, 출결, 태스크 상태를 볼 수 있다.
- 자신이 참여 중인 프로젝트의 전체 내용을 볼 수 있다.
- 자신이 참여하지 않은 프로젝트는 기본적으로 전체 열람 불가다.
- 지도학생의 프로젝트 없는 기록 중 공개범위가 `advisor` 이상인 내용은 볼 수 있다.

#### 학생
- 자신의 데이터 전체 관리 가능
- 자신이 참여한 프로젝트 데이터 열람 가능
- 다른 학생의 프로젝트 밖 개인 데이터는 열람 불가

#### 외부업체
- 자신이 참여한 프로젝트 범위 안에서만 열람 가능
- 해당 프로젝트의 데일리, 태스크, 이슈, 일정, 성과물은 열람 가능
- 프로젝트 범위 안의 블록에는 댓글 작성 가능
- `advisor`, `private`, `internal` 블록에는 접근 및 댓글 작성 불가
- 프로젝트 밖 개인 업무, 행정, 장학금, 사적 일정은 열람 불가

## 4. 핵심 도메인 개념

이 시스템은 아래 네 가지 흐름이 중심이다.

- 사람 기준 관리: 교수 → 지도학생, 학생 개인 활동, 출결, 일정
- 프로젝트 기준 관리: 프로젝트 멤버, 태스크, 데일리, 일정
- 데일리 기반 운영: 자유롭게 작성하되 나중에 구조화해서 본다
- 권한별 집계: 교수, 학생, 외부업체, 관리자마다 다른 관점으로 본다

---

## 5. 데일리 시스템 설계

데일리 시스템은 이 제품의 핵심이다.

### 5.1 UX 목표
- 최대한 쓰기 쉬워야 한다.
- 작성 흐름이 끊기면 안 된다.
- 사용자가 처음부터 프로젝트, 태그, 공개범위를 일일이 고르게 만들지 않는다.
- 대신 작성 후 문단 또는 블록 단위로 구조화할 수 있게 한다.

### 5.2 작성 방식
데일리는 `단일 에디터 화면`에서 작성한다.

사용자는 자유롭게 내용을 쭉 작성할 수 있다. 이후 특정 문단 또는 블록을 선택해서 아래 속성을 붙일 수 있다.

- 프로젝트 연결
- 섹션 지정
- 공개 범위 지정
- 태그 지정

### 5.3 블록 단위 구조화
초기 구현에서는 문장 일부 선택이 아니라 `블록 단위` 편집을 기준으로 한다.

블록 단위란 아래를 의미한다.
- 줄바꿈이나 문단 기준으로 나뉜 내용 단위
- 사용자가 특정 블록을 선택할 수 있음
- 선택한 블록에 프로젝트, 태그, 섹션, 공개범위를 부여 가능

### 5.4 블록 섹션 종류
각 블록은 다음 섹션 중 하나를 가진다.
- `yesterday`
- `today`
- `issue`
- `misc`

### 5.5 공개 범위
각 블록은 공개 범위를 가진다.
- `private`
- `advisor`
- `internal`
- `project`

### 5.6 internal 범위 확정
v1에서는 `internal`을 `전체 내부 사용자`로 확정한다.
외부업체 사용자는 `internal` 블록을 볼 수 없다.
향후 필요 시 `lab` 또는 `organization_unit` 개념을 추가해 랩 단위 공개로 확장한다.

### 5.7 기본 공개 범위 규칙
- 프로젝트 연결 블록: 기본값 `project`
- 프로젝트 없는 블록: 기본값 `advisor`
- 민감한 블록은 `private`로 낮출 수 있어야 한다.

### 5.8 원문과 블록의 동기화 전략
- 초기 자유 입력은 `raw_content`에 저장한다.
- 저장 또는 구조화 시점에 `daily_blocks`를 생성한다.
- 구조화 이후 실질 source of truth는 `daily_blocks`다.
- 최종 표시 내용은 블록 기준으로 재조합한다.
- `raw_content`는 원문 백업 성격으로 유지한다.
- 원문 모드에서 전체 재편집 시 블록 재생성 흐름을 허용하되, 기존 구조화 정보가 덮어써질 수 있음을 UI에서 고지한다.
- v1에서는 raw_content와 block의 완전 양방향 동기화를 하지 않는다.

### 5.9 검색 및 필터링 전략
v1 검색은 PostgreSQL 기반으로 구현한다.
지원 필터:
- 날짜 범위
- 작성자
- 프로젝트
- 태그
- 섹션
- 이슈만 보기
- 텍스트 검색

텍스트 검색은 PostgreSQL full-text search 또는 ILIKE 기반으로 시작한다.
별도 검색 엔진은 초기 범위에서 제외한다.

### 5.10 임시저장 및 복구
데일리 작성 중 네트워크 문제나 실수로 인한 데이터 손실을 줄이기 위해 v1에서 아래를 고려한다.

- 브라우저 로컬 저장소 기반 임시저장
- 일정 주기 자동 저장
- 저장 실패 시 사용자에게 복구 가능 상태 안내
- 재진입 시 마지막 draft 복원 제안

완전한 오프라인 우선 앱은 v1 범위에서 제외하지만, 데일리 작성 경험 보호를 위해 draft 복구는 Phase 2에 포함하는 것이 좋다.

### 5.11 댓글
댓글은 데일리 전체가 아니라 `블록 단위`로 단다.

## 6. 프로젝트 시스템 설계

### 6.1 프로젝트 목표
프로젝트는 협업의 기본 단위다.
프로젝트별로 아래 정보가 연결되어야 한다.
- 멤버
- 역할 및 권한
- 태스크
- 데일리 블록
- 일정
- 첨부 파일
- 리포트
- 외부업체 협업 정보

### 6.2 프로젝트 리드
프로젝트 리드는 교수만 가능한 것이 아니다. 교수, 학생, 외부업체 사용자 모두 프로젝트 리드가 될 수 있다.

### 6.3 프로젝트 권한
프로젝트 권한은 최소 아래 구분을 둔다.
- `viewer`
- `member`
- `manager`
- `lead`

---

## 7. 태스크 시스템 설계

### 7.1 기본 원칙
태스크는 공동 할당 가능한 구조로 설계한다.

### 7.2 상태 추천
- `todo`
- `in_progress`
- `blocked`
- `review`
- `done`

---

## 8. 출결 시스템 설계

### 8.1 기본 방식
출결은 `수동 체크인 / 수동 체크아웃` 방식으로 구현한다.

### 8.2 자동화는 초기 범위에서 제외
데일리 작성 시 자동 출근 처리 같은 방식은 초기에 도입하지 않는다.

---

## 9. 캘린더 및 일정관리 설계

캘린더는 핵심 기능으로 포함한다.
학생은 프로젝트 일정, 개인 일정, 수업 일정, 회의 일정, 발표/마감 일정을 본다.
교수는 학생별 일정과 업무 상태를 함께 본다.

---

## 10. 태그 정책

태그는 `전역 공용 태그 + 프로젝트 전용 태그`를 함께 지원한다.
프로젝트는 태그가 아니라 직접 연결한다.

---

## 11. 알림 시스템 설계

v1 알림은 `인앱 알림 중심`으로 구현한다.
- 인앱 알림: 필수
- Rocket.Chat 웹훅: 일부 이벤트에서 선택 적용 가능
- 이메일: v1 제외

우선 알림 이벤트:
- 태스크 배정
- 데일리 이슈 댓글
- 체크아웃 누락
- 일정 리마인더
- 리포트 발행
- SOTA 배정

---

## 12. 첨부 및 파일 전략

v1 파일 정책은 `링크 중심 + 소형 파일 업로드 허용`으로 한다.
- 대용량 영상, 데이터셋, 산출물 원본은 앱 내부 저장 대신 NAS/외부 스토리지 링크 우선
- 앱 내부 업로드는 이미지, PDF, 문서 등 경량 파일 위주
- v1 프리뷰는 이미지/PDF/외부 링크 열기 수준
- 저장소는 추후 MinIO 또는 S3 호환 구조로 확장 가능하게 설계

---

## 13. SOTA 시스템 설계

SOTA는 추천, 배정, 검수, 제출 추적, 리뷰 요약까지 지원한다.

---

## 14. 리포트 설계

리포트는 단일 문서 하나가 아니라 권한별/필터별 조회 뷰 + 요약 생성으로 설계한다.

---

## 15. 감사 로그(Audit Log)

권한이 복잡하고 외부업체가 참여하므로 감사 로그를 둔다.
v1에서는 `중요 변경 행위` 중심으로 기록한다.
필수 기록 대상:
- 사용자 역할 변경
- 지도교수 관계 변경
- 프로젝트 멤버 추가/제거
- 프로젝트 권한 변경
- 데일리 블록 공개범위 변경
- 태스크 배정 변경
- 중요 일정 생성/수정/삭제

- Infra: Ubuntu Server + Docker Compose + Cloudflare Tunnel
- Future AI/LLM: OpenClaw + H100 (별도 머신)

## 18.1 인프라 및 네트워크 설계

### 서버 환경
- 고정 IP를 부여받은 물리 PC 1대 (Ubuntu Server)
- 해당 PC에서 Docker Compose로 전체 서비스를 운영한다

### Docker Compose 구성
```
services:
  db:        PostgreSQL
  backend:   FastAPI (uvicorn)
  frontend:  React (nginx 또는 serve)
  # Phase 4에서 추가:
  rocketchat: Rocket.Chat
  mongo:      MongoDB (Rocket.Chat용)
```

### 네트워크 접근 구조
- **외부 접속**: Cloudflare Tunnel을 통해 외부 도메인(예: `hub.example.com`)으로 HTTPS 접속
- **내부 접속**: 같은 네트워크 내에서도 동일한 외부 도메인으로 접속 (Tunnel 경유)
- **도메인 통일**: 내부/외부 모두 하나의 도메인을 사용한다

### 도메인 통일 원칙
내부 사용자도 Cloudflare Tunnel 도메인을 사용한다. 이유:
- Google OAuth2 redirect URI가 하나로 통일된다
- SSL 인증서를 별도 관리할 필요가 없다 (Cloudflare 자동 처리)
- CORS 설정이 단순해진다
- 내부/외부 사용자 경험이 동일하다

### SSL/TLS
- 외부 구간: Cloudflare에서 자동 HTTPS
- Tunnel ↔ 서버 구간: Cloudflare Tunnel이 로컬 HTTP 포트로 연결 (서버 내부는 HTTP 허용)
- 자체 인증서 관리는 v1에서 불필요

### Google OAuth2 설정
- redirect URI: `https://hub.example.com/api/v1/auth/callback`
- 단일 도메인이므로 OAuth 앱 설정이 하나로 끝난다

### H100/OpenClaw 연동 (Phase 4)
- H100은 별도 물리 머신에서 운영
- Hub 서버에서 H100 머신으로 내부 네트워크 API 호출
- 연동 방식은 Phase 4에서 상세 설계

### Rocket.Chat (Phase 4)
- 같은 PC에 Docker 컨테이너로 추가
- Cloudflare Tunnel에 서브도메인 추가 (예: `chat.example.com`)
- Google SSO를 Hub과 공유 가능

### 포트 구성 (Docker 내부)
- `5432`: PostgreSQL
- `8000`: FastAPI
- `3000`: React dev / nginx
- Cloudflare Tunnel이 외부 → 로컬 포트로 프록시

## 19. API 설계 원칙

v1 API는 `REST` 기준으로 설계한다.
- base path: `/api/v1`
- JSON 응답 사용
- 목록 API는 페이지네이션 필수
- 권한 검사는 백엔드에서 최종 수행
- 초기 페이지네이션은 `page/limit` 권장
